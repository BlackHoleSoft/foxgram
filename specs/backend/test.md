# Тесты для Backend API

## Общее
- Все тесты должны быть написаны с использованием Jest + supertest
- Перед каждым тестом должен быть setup тестовой БД
- После каждого теста должен быть cleanup тестовых данных

## Тесты для POST /api/v1/register

### Успешная регистрация нового пользователя
- Запрос с валидным username должен возвращать 200
- Ответ должен содержать guid пользователя
- В БД должна создаться запись с пользователем

### Регистрация пользователя с существующим guid
- Если случайно сгенерированный guid уже существует, система должна перегенерировать и повторить попытку
- Должен быть огранич на количество попыток перегенерации = 3
- Если после 3 попыток guid не найден, должен возвращаться ошибка

### Регистрация с невалидными данными
- Пустой username должен возвращать 400
- username с недопустимыми символами должен возвращать 400
- username слишком длинный должен возвращать 400
- Отсутствие поля username должно возвращать 400

### Регистрация с дубликатом username
- Два пользователя с одинаковым username должны иметь разные guid
- Это не должно вызывать ошибку

## Тесты для POST /api/v1/messages/send

### Успешная отправка сообщения
- Запрос с валидными sender, receiver, payload должен возвращать 200
- Должен создаваться файл с сообщением в messages/
- Должна создаваться запись в БД message
- create_date должен быть установлен в текущую дату

### Отправка сообщения несуществующему пользователю
- Если sender или receiver не существуют в БД, должен возвращаться 404

### Отправка сообщения самому себе
- Должна позволять отправку сообщения самому себе
- Должен создаваться файл и запись в БД

### Отправка пустого сообщения
- Пустой payload должен возвращать 400

### Отправка сообщения с очень длинным текстом
- Сообщение максимальной длины должно обрабатываться корректно
- Слишком длинное сообщение должно возвращать 400

### Отправка сообщения с недопустимыми символами в payload
- Специальные символы в payload должны сохраняться корректно
- Unicode символы должны обрабатываться корректно

### Отправка сообщения с невалидными guid
- Неверный формат sender guid должен возвращать 400
- Неверный формат receiver guid должен возвращать 400
- Отсутствие обязательных полей должно возвращать 400

## Тесты для POST /api/v1/messages/get

### Успешное получение сообщений
- Запрос с валидным receiver_guid должен возвращать 200
- Ответ должен содержать total и messages массив
- Количество возвращенных сообщений не должно превышать count
- Должны возвращаться только сообщения для указанных receiver и sender

### Получение сообщений с пагинацией
- При start=0, count=10 должны возвращаться первые 10 сообщений
- При start=10, count=10 должны возвращаться сообщения со 11 по 20
- При start > общего количества сообщений должен возвращаться пустой массив

### Получение сообщений для несуществующего пользователя
- Если receiver_guid или sender_guid не существует, должен возвращаться пустой массив

### Получение сообщений с невалидными параметрами
- start < 0 должен возвращать 400
- count < 1 должен возвращать 400
- count > 1000 должен возвращать 400 (максимальное лимит)
- Отсутствие receiver или sender (хотя бы одного из параметров) должно возвращать 400

### Получение сообщений в правильном порядке
- Сообщения должны быть отсортированы по дате создания (от новых к старым)
- create_date в ответе должен соответствовать дате из БД

## Тесты для GET /api/version

### Успешное получение версии
- Запрос должен возвращать 200
- Ответ должен содержать version из package.json
- Version должен быть в формате X.Y.Z

## Тесты безопасности

### SQL инъекции
- Попытка SQL инъекции в username должна блокироваться
- Попытка SQL инъекции в message payload должна блокироваться

### XSS атаки
- Попытка XSS в username должна экранироваться
- Попытка XSS в message payload должна экранироваться
